## 1. Цель документа

Incident Response определяет, **как реагировать на инциденты** — события, угрожающие капиталу или целостности системы.

Принципы:
1. Сохранность капитала
2. Контроль ситуации
3. Фиксация фактов
4. Коммуникация

**Ни один инцидент не требует немедленного героического действия.** Большинство инцидентов лучше переждать с холодной головой, чем реагировать на adrenaline.

---

## 2. Типы инцидентов

- **Smart Contract Incident** — эксплойт или баг whitelisted протокола
- **Liquidity Incident** — резкое падение ликвидности
- **Data Incident** — расхождение данных, oracle failure, RPC outage
- **Execution Incident** — failed transaction, неверный адрес, газ-проблемы
- **AI / Automation Incident** — агент действовал вне политик, prompt injection, coordination failure
- **Regulatory Incident** — попадание whitelisted протокола в санкционный список
- **Access Incident** — компрометация ключей, утечка credentials

### 2.0. Severity Levels

| SEV | Описание | SLA реакции |
|-----|----------|-------------|
| **SEV-1** | Угроза > 5% капитала или полная потеря доступа | **1 час** |
| **SEV-2** | Угроза 1–5% капитала или частичная деградация | **24 часа** |
| **SEV-3** | Угроза < 1% или операционные неудобства | **7 дней** |
| **SEV-4** | Минимальный импакт, для статистики | weekly cycle |

Сomбинированные инциденты — берётся максимальная Severity.

## 3. Общие процедуры

### 3.1. Triage

При обнаружении любого инцидента:
1. **Зафиксировать факт** в `incidents.log` (timestamp, что произошло, известные факты).
2. **Классифицировать** Severity по доступной информации (предпочтительно — консервативно, повышая на 1 ступень при неопределённости).
3. **Уведомить** соответствующих лиц (на Self-Capital — Owner себе).
4. **Защитное действие** — если позволяет Severity SLA.
5. **НЕ принимать необратимых решений** в первые 30 минут (кроме SEV-1 с явной экспозицией).

### 3.2. Запрещённые эмоциональные паттерны

При инциденте **запрещены**:
- **Panic-sell** — паническая продажа при первых признаках проблемы;
- **Revenge trading** — попытка отыграться после потери;
- **FOMO** — гонка за упущенной возможностью;
- **Anchoring** — привязка к цене входа («продам когда восстановится»);
- **Doubling down** — увеличение позиции при просадке.

Если поймал себя на этом — **остановиться**, отойти, вернуться через час.

## 4. Процедуры по типам инцидентов

### 4.1. Smart Contract Incident
См. Risk Policy 7.1. Различать активный и завершённый эксплойт.

### 4.2. Liquidity Incident
См. Risk Policy 7.2.

### 4.3. Data Incident
- два независимых источника расходятся → degraded data;
- три расходятся → broken data → safe-mode на затронутую часть.

### 4.4. Execution Incident
- failed transaction → анализ причины, retry только при ясной причине;
- неверный адрес → SEV-1, не корректирующая транзакция (адрес уже получил средства).

### 4.5. AI / Automation Incident
- агент сделал что-то вне политик → немедленный autopilot stop, full review логов;
- prompt injection обнаружен → sandbox-тестирование всех LLM-агентов;
- coordination failure → проверить anti-loop guards (Agent Architecture 5.2).

### 4.6. Yearn Exit Procedure (добавлено в v0.4.5)

Триггеры для exit из Yearn V3 yvUSDC:
- обнаружение leveraged looping в стратегиях vault (на основе on-chain или ChainSecurity audit);
- изменение vault strategy без 48ч timelock;
- ymultisig governance proposal по добавлению exotic стратегий.

Severity: SEV-2 (если только обнаружено) или SEV-3 (если ещё нет implementation).

Варианты redistribution:
- **A:** растворить в Tier 1 (увеличить Aave/Compound/Sky пропорционально);
- **B:** временно в Sky sUSDS до следующего ADR.

Cost considerations: Yearn вход дорогой, exit — нормальный. Перед exit оценить gas против ожидаемого ущерба.

## 6. Post-mortem

Обязательный post-mortem для SEV-1 и SEV-2 в течение 7 дней:
- что произошло (факты);
- как обнаружено;
- что сделано;
- что было сделано правильно;
- что было сделано неправильно;
- что нужно изменить (action items с дедлайнами);
- ADR (если применимо).

### 6.1. Закрытие инцидента

Инцидент закрывается в одном из трёх статусов:
- **Подтверждённый** — реальный инцидент, проведён post-mortem, action items закрыты;
- **Ложный** — false positive, документируется причина false positive (для калибровки порогов);
- **Частично подтверждённый** — реальный, но менее severe чем классифицировано; пересмотр Severity.

---

## 7. Tabletop Exercises

**Квартально** — упражнение по реакции на гипотетический инцидент:
- Owner выбирает сценарий (например: «Aave V3 exploit на $500M», «Tornado Cash precedent на Sky»);
- проходит процедуру triage и реакции, как если бы инцидент был реальный;
- результат фиксируется в quarterly report;
- выявленные пробелы → ADR / правки в политиках.

---

## 8. Условия выхода в v1.0

Incident Response переходит в v1.0 после того как:
- проведено минимум 2 tabletop exercises;
- обработан минимум 1 реальный инцидент (даже SEV-4) с полным post-mortem;
- проверены SLA для всех SEV уровней.
