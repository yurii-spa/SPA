# Протокол оркестратора (ENV_SETUP_BRIEF_v3 · Этап 3)

> Точный, упорядоченный чеклист, который оркестратор (headless-сессия Claude Code на
> LaunchAgent, либо ручной прогон) исполняет **каждый цикл**. Механические шаги — через
> детерминированный `scripts/orchestrator_queue.py`; шаги-суждения — сам оркестратор.
> Источник правды — файлы в git. LLM здесь допустим (это project-management, НЕ
> risk/execution/monitoring — там LLM по-прежнему запрещён).

## ▶️ НАЧАЛО ЦИКЛА

**Шаг 0 — контекст.** Прочитать `docs/STATE.md`, `docs/decisions/INDEX.md`,
`docs/SYSTEM_BRIEFING.md`. Свериться с открытыми вопросами и активными задачами.

**Шаг 1 — разбор Inbox (Этап 6).**
Сначала втянуть свободные заметки Obsidian из `inbox/` в карточки:
`python3 scripts/orchestrator_queue.py ingest-notes` (каждая заметка → `inbox`-карточка source=obsidian,
оригинал уезжает в `inbox/.ingested/`). Затем: `python3 scripts/orchestrator_queue.py list --type inbox --status new --json`.

Входы, все дают одинаковые `inbox`-карточки: карточка в Nimbalyst · заметка в `inbox/` (Obsidian) ·
Telegram `/task <текст>` · Telegram голосовое (whisper-расшифровка офлайн, `source: voice`).

Для каждой `new` inbox-карточки классифицировать (амендмент §6.3):
- **ЗАДАЧА** (есть конкретное действие) → создать задачу с acceptance-критериями (см. «Правила задач»),
  перевести карточку в `in-progress` (`set-status <path> in-progress`), вписать ссылку на порождённую
  задачу. Telegram-ответ владельцу: «создал задачу: …».
- **ИДЕЯ** (мысль / наблюдение / «а что если») → сохранить в `docs/ideas/<дата-slug>.md` с датой,
  **задачу НЕ создавать** (инвариант «идея ≠ инструкция»). Telegram-ответ: «записал как идею».
- **НЕПОНЯТНО** или **конфликт с инвариантами** (risk-политика, refusal-first, no-solicitation …) →
  создать карточку `own-*` `needs-owner` (`create --type owner-decision …`, формат §2.4) с уточняющим
  вопросом / объяснением конфликта + `notify`, **не исполнять молча**. Telegram-ответ: «есть вопрос — смотри карточку».
- **§6.4 — закрыть исходную карточку со ссылками:** после обработки допиши в тело inbox-карточки ссылки
  на порождённые задачи/идею/`own-*` и переведи её в `done` (`set-status <path> done`) — владелец всегда
  проследит судьбу задания.

**Шаг 1b — промоушен идей/черновиков (Этап 7.3).**
`python3 scripts/orchestrator_queue.py promotions --json` — найти заметки с тегом `#promote` в
`docs/ideas/` и `docs/rules-draft/`. Для каждой: превратить в подходящую форму —
**правило** (`.claude/rules/` / `CLAUDE.md`), **ADR** (`docs/decisions/`) или **задачу-карточку**
(`create --type inbox`). Затем пометить исходник обработанным: заменить `#promote` → `#promoted-<дата>`
и дописать ссылку на созданное. **Без `#promote` — не действовать** (идея ≠ инструкция).

**Шаг 2 — инжест решений владельца.**
`python3 scripts/orchestrator_queue.py list --type owner-decision --status owner-done --json`.
Для каждой `owner-done` карточки:
1. Прочитать ответ владельца в теле карточки.
2. Исполнить / декомпозировать решение (по «Правилам задач»). Если конфликтует с инвариантом —
   НЕ исполнять, вернуть карточкой `needs-owner` с объяснением.
3. Если это архитектурное решение — выписать/дополнить ADR (`docs/decisions/`, шаблон `_TEMPLATE.md`),
   добавить строку в `docs/decisions/INDEX.md`.
4. Перевести карточку в `ingested`:
   `python3 scripts/orchestrator_queue.py set-status <path> ingested`.
   **owner-done ставит ТОЛЬКО владелец — инструмент запрещает агенту (инвариант #14).**
5. Добавить строку в раздел «Последние решения» `docs/STATE.md`.

## ⏹️ КОНЕЦ ЦИКЛА

**Шаг 3 — обновить память.** Обновить `docs/STATE.md` (фокус, активные задачи, последние решения,
открытые вопросы) и дописать `docs/journal/<ISO-неделя>.md` (что сделано за цикл).

**Шаг 4 — новые вопросы владельцу.** Только карточкой `own-*` `needs-owner` по формату §2.4
(**Контекст / Инструкция / Критерий «сделано» / Что произойдёт после**), затем обязательно
уведомить (§3.3): `python3 scripts/orchestrator_queue.py notify <path>`.

## 📏 Правила задач (§3.4)

Каждая техническая задача:
- **acceptance criteria** прописаны в теле;
- работа в **изолированном git worktree** (не пачкать основную ветку параллельных процессов);
- **тесты до закрытия** (код-изменения без зелёных тестов не закрываются).

## 🚦 Инварианты протокола

1. Агент **никогда** не ставит `owner-done` (только владелец; `set_status` бросает `OwnerDoneForbidden`).
2. Конфликт задания с risk-инвариантами → `needs-owner`, не исполнять молча.
3. Идея ≠ инструкция: `docs/ideas/` и `docs/rules-draft/` не исполняются без промоушена (`#promote` / явная карточка).
4. Новые вопросы владельцу — только карточки `needs-owner` + Telegram-уведомление.
