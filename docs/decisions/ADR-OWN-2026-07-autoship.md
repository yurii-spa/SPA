# ADR-OWN-2026-07 · Автономный авто-шип сайта (owner-approved)

- **Статус:** ACCEPTED
- **Дата:** 2026-07-15
- **Владелец:** Yurii (явное согласие в сессии, AskUserQuestion «Как автономный оркестратор
  должен менять живой сайт?» → выбран вариант **«Полный авто-шип (как в плане)»**).

## Контекст

Владелец передумал держать процесс attended и хочет, чтобы система работала автономно
(«отрабатывала мой пакет»). Для сайта это означает: автономный оркестратор (`com.spa.orchestrator`)
может САМ пушить «безопасные» правки сайта в live (earn-defi.com, push==live через Cloudflare Pages),
без ручного клика владельца на каждую.

## Решение

**Полный авто-шип РАЗРЕШЁН** под owner-gate линтером:
- Оркестратор пушит SAFE-правки сайта (layout/CSS, компонент-рефакторы, не-legal копирайт, SEO,
  багфиксы, динамические чтения чисел) в live автоматически.
- OWNER-GATED классы — **числа доходности, нейминг тиров, расшифровка «SPA», legal/disclaimer,
  solicitation, удаление honesty-токенов** — НИКОГДА не уезжают сами: линтер их ловит (exit 2),
  `safe_site_push.py` создаёт карточку `needs-owner` + notify, пуш НЕ происходит.
- Kill-switch: `SPA_ORCHESTRATOR_ARMED=0` (или `bootout`) выключает всю автономию.

## Явно принятый риск (владелец проинформирован)

In-repo owner-gate — **кооперативный** контроль (механизирует мандат оркестратора, ловит честный
дрейф/ошибки), НЕ криптозащита: та же машина теоретически может его обойти. Владелец это прочитал
(вариант прямо называл «гейт кооперативный, не бронебойный») и выбрал полный авто-шип. Более
строгая опция (branch-protect / preview + ручной промоушен) остаётся доступной как будущее
ужесточение, если потребуется.

## Как проверяется

- `scripts/check_owner_gate.py` red-team: подмена числа тира `20%→30%` → GATED (exit 2); чистая
  layout-правка → CLEAN (exit 0). Оба зелёные на момент принятия.
- Hard interlock в `push_to_github*.py` (активен только при `SPA_AUTONOMOUS=1`) — автономный пуш в
  `landing/` без прохождения `safe_site_push.py` отклоняется (fail-closed).
