## 📋 EXECUTIVE SUMMARY

Проект SPA (DeFi Yield Optimizer) с нотальным прогрессом (553 задачи, 313 analytics-модулей)
страдает от **системных governance-проблем**, которые циклически приводят к одним и тем же ошибкам.

"добро" на следующую кирпичную кладку, но не знает что уже построено, что заложено в фундамент,
и какие стены несущие.

## 🔬 ЧАСТЬ 2: ROOT CAUSE ANALYSIS

### RC-001: Отсутствие единого источника истины для оперативного состояния
**Симптом:** 4 файла (MEMORY.md, RULES.md, CURRENT_STATE.md, KANBAN.json) говорят разное о done_count и sprint_current.

**Причина:** Агенты обновляют разные файлы в разном темпе. KANBAN.json обновляется каждым спринтом (аддитивно). MEMORY.md обновлялся редко и устарел. CURRENT_STATE.md отстал на 1+ версию.

**Следствие:** Новый агент, полагающийся на MEMORY.md, думает что сделано 183 задачи, когда реально 553. Он может создавать задачи, уже выполненные другими агентами.

**Фикс:** Заморозить MEMORY.md. Сделать CURRENT_STATE.md единственным читаемым summary-файлом, синхронизированным с KANBAN.json. Правило: агент читает KANBAN.json для точных цифр, CURRENT_STATE.md для оперативного контекста.

---

### RC-002: Отсутствие Pre-Work Checklist — агенты стартуют "вслепую"
**Симптом:** Агенты создают дубликаты модулей (15+ групп файлов с похожими именами), принимают решения на основе устаревшего контекста.

**Причина:** Нет обязательного протокола чтения перед стартом работы. RULES.md имел Startup Protocol, но в разделе "Правила", не как первый экран.

**Следствие:** yield_curve имеет 5 файлов вместо 1. capital_efficiency — 4. cross_chain — 4. При 313 модулях вероятность дублей ~40%.

**Фикс:** DEVELOPMENT_RULES.md §Pre-Work Checklist — 7 обязательных шагов, первый из которых "ls spa_core/analytics/ | grep <keyword>".

---

### RC-003: Потеря архитектурных решений между итерациями
**Симптом:** Одни и те же решения принимаются повторно. Пример: "никогда не встраивать PAT" — было в RULES.md с 2026-06-10, но агенты продолжали рисковать.

**Причина:** DECISIONS.md содержит только 3 записи при 200+ спринтах. RULES.md содержит правила, но его не читают первым. Нет структурированного ADR-реестра в машиночитаемом формате.

**Следствие:** Повторное обсуждение уже решённых вопросов. Повторные инциденты.

**Фикс:** AI_ASSISTANT_RULES.md как обязательный первый файл каждой сессии. Все принятые решения с немедленной записью в DECISIONS.md как часть DoD.

---

### RC-004: Push-система создаёт иллюзию доставки при реальном лаге
**Симптом:** GitHub репо периодически stale. Пользователь не знает, что запушено, а что нет.

**Причина:** 
1. autopush не установлен (MP-313 открыт с 2026-06-10, 3+ дня не решён)
2. 200+ push-скриптов создаются, но нет гарантии что они запускаются
3. `.push_log` существует в run_all_pushes.sh, но агенты не проверяют его состояние
4. KANBAN.json показывает `delivery_status: shipped_local` — не `shipped_remote`

**Следствие:** done_count=553 но в GitHub может быть 400. Track record не синхронизирован.

**Фикс:** GIT_WORKFLOW.md + KNOWN_ISSUES.md KI-005. USER ACTION: bash run_all_pushes.sh.

---

### RC-005: Параллельные спринты создают race conditions в KANBAN.json
**Симптом:** done_count расходится (371 в columns.done + 24 в tasks = 395, но done_count=553). sprint_log содержит только 5 записей при 600+ спринтах.

**Причина:** 2 агента работают параллельно. Оба читают done_count=N, оба пишут done_count=N+1, итого прирост +1 вместо +2. Аналогично sprint_log — один агент перезаписывает другого.

**Следствие:** Статистика неточна. Sprint history потеряна.

**Фикс:** DEVELOPMENT_RULES.md §Параллельные спринты — аддитивный инкремент + перечитывать файл перед записью. Принять расхождение done_count ±5 как норму.

---

### RC-006: KANBAN.json деградирует в неструктурированный лог
**Симптом:** 80+ ключей `_vNNN_dispatch_note` в корне JSON. Реальные данные (tasks, columns) смешаны с мета-информацией.

**Причина:** Каждый спринт добавляет dispatch_note в корень, не в sprint_log. Размер KANBAN.json неконтролируемо растёт. Парсинг неэффективен.

**Следствие:** KANBAN.json трудно читать программно. При агрессивном параллелизме — риск конфликтов.

**Фикс (долгосрочный):** Мигрировать dispatch_notes в data/sprint_notes.json. Текущее состояние: ПРИНЯТО как технический долг (KI-007).

---

### RC-007: Агенты просят пользователя о том, что сами могут сделать
**Симптом:** "Пожалуйста, обновите KANBAN.json", "Создайте тест-файл".

**Причина:** Недостаточно чёткая граница — что агент делает сам vs что требует пользователя.

**Следствие:** Пользователь перегружен запросами. Задачи накапливаются.

**Фикс:** AI_ASSISTANT_RULES.md §Разрешённые действия без подтверждения vs §Действия только с запросом. Чёткая граница: агент делает всё кроме пуша в GitHub и изменений в Keychain/launchd.
