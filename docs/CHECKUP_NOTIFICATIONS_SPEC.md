## 0. Governing constraint (почему это не «просто рассылка»)

Весь moat DeFi Checkup — **честность**: детерминированность, fail-closed, non-custodial, evidence-tagged
(L0–L6), **никаких выдуманных чисел**, **никакого LLM в risk-пути**, «мы измеряем риск, а не советуем
сделку и не держим ключи» (`docs/CHECKUP_REPORT_10OF10.md`). Уведомления НЕ должны это ломать:

- **Никакого solicitation / trade-советов** в письме или сообщении. Уведомление сообщает *факт об
  изменении риска*, а не «продай $170k». Кнопок execute / Safe-transaction — нет (инвариант: AI не
  подписывает, ключей не держит).
- **Только измеренные числа.** Materiality-порог считается от реального `Exit-NAV $` /
  `capital-under-elevated-risk $`, которые движок уже вычисляет. Ни одного фабрикованного composite-score
  (правило «no health score»), ни вероятностей сценариев.
- **Fail-closed.** Нет свежих данных по кошельку / фид недоступен → шлём «не смогли проверить, данные
  устарели», а НЕ «всё спокойно». Тишина ≠ безопасность.
- **Secrets только из окружения** (Railway env / Keychain), никогда в файлах (инцидент PAT-leak 2026-06-10).
- **Двойное согласие + отписка.** E-mail — double opt-in (own-07: `RESEND_API_KEY` — owner-gated, письма
  выключены до включения владельцем). Каждое письмо — one-click unsubscribe (CAN-SPAM/GDPR-гигиена).

## 1. Каналы

| Канал | Роль | Транспорт | Секрет | Статус |
|---|---|---|---|---|
| **E-mail** | база пользователей, дайджест, «важное» | Resend API | `RESEND_API_KEY` | ⛔ owner-gated (own-07) |
| **Telegram** | мгновенный пинг по материальным событиям, opt-in bind | Bot API (per-user chat_id) | `TELEGRAM_*` (Keychain на хосте SPA; для checkup — отдельный бот-токен в Railway env) | 🟡 build |
| Push (браузер / mobile) | — | — | — | 📦 backlog (см. §7) |
| Webhooks (B2B) | — | — | — | 📦 backlog (см. §7) |

**Привязка Telegram к кошельку/отчёту:** пользователь на `/check` жмёт «получать оповещения» → генерируется
одноразовый `bind_token` (короткоживущий, HMAC от `wallet_ref + salt`, salt из env — ср. own-07
`WALLET_REF_SALT`) → deeplink `https://t.me/<checkup_bot>?start=<bind_token>` → бот связывает `chat_id` с
watchlist-записью. **Никаких адресов кошельков в открытом виде в Telegram** — только `wallet_ref` (хеш).

## 2. Триггеры и порог по $-материальности

Уведомление рождается только при **изменении между двумя отчётами** одного кошелька (движок уже считает
дельты — `computeReportDelta`, DC-16). Материальность — по **абсолютному $ капитала под риском**, не по
косметике.

### 2.1 Классы событий (детерминированные, keyed на `reason_code`)

| Класс | Условие (пример) | Канал по умолчанию |
|---|---|---|
| **CRITICAL** | появился Critical-риск (depeg/oracle/bridge на held-позиции), ИЛИ `capital-under-elevated-risk` вырос ≥ **$M_high** | Telegram мгновенно + e-mail |
| **IMPORTANT** | новый Important-риск, ИЛИ рост риск-капитала ≥ **$M_mid** и < $M_high, ИЛИ падение coverage ниже порога | e-mail (Telegram по opt-in) |
| **WATCH** | Watch-риск / мелкая дельta ≥ **$M_low** | только weekly digest |
| **RESOLVED** | ранее поднятый риск ушёл (дельта в плюс) | weekly digest (в CRITICAL-письме — строкой «что улучшилось») |
| **DATA-STALE** | не смогли переоценить кошелёк N дней (фид/скан недоступен) | e-mail (fail-closed, «проверка устарела») |

### 2.2 Пороги материальности (`$M_*`) — ПАРАМЕТРЫ, не хардкод

- Пороги задаются per-user (по умолчанию — глобальные дефолты), хранятся в БД watchlist-записи.
- **Дефолты (предложение, owner может поменять):** `$M_low = $1,000` · `$M_mid = $10,000` ·
  `$M_high = $100,000` **ИЛИ** относительные `1% / 5% / 20%` от NAV кошелька — берём **max(абсолют, относит.)**,
  чтобы и киту, и мелкому было по делу.
- Порог `#16 client Risk-Policy` (roadmap): в перспективе пользователь/казна задаёт свои
  `max_concentration / min_TVL / banned_protocols` → триггер считается против **его** политики. Пока — дефолты.
- **Anti-spam / дебаунс:** не чаще 1 мгновенного пинга на кошелёк за `T_debounce` (предложение — 6 ч);
  всё остальное копится в weekly digest. Один и тот же `reason_code` не пингуется повторно, пока не RESOLVED
  и снова не поднят (гистерезис — избегаем flapping на границе порога).

## 3. Deeplink-схема (мгновенная маршрутизация по материальности)

Каждое уведомление ведёт прямо в нужное место отчёта, а не на общую главную.

```
https://checkup.earn-defi.com/check/<wallet_ref>?report=<report_id>&focus=<reason_code>#<anchor>
```

- `<wallet_ref>` — хеш, не сырой адрес.
- `report=<report_id>` — конкретный снапшот отчёта (иммутабельный, с `output_hash` — привязка к `/verify`).
- `focus=<reason_code>` — какой именно риск подсветить (движок раскрывает соответствующую карточку).
- `#<anchor>` — секция (`#exit-liquidity`, `#concentration`, `#approvals`, …).
- **По материальности:** CRITICAL-диплинк открывает отчёт сразу на топ-1 риске (Critical/Important/Watch
  bucket, top-3 first — как в отчёте); WATCH-диплинк из дайджеста — на общий обзор.
- **Telegram-диплинк** дублируется inline-кнопкой «Открыть отчёт»; текст сообщения самодостаточен (вердикт +
  $-число), кнопка — для деталей.
- Диплинки **не содержат секретов/PII** и не логинят автоматически (non-custodial: отчёт публично-проверяем
  по `wallet_ref`, приватные действия — за отдельным opt-in).

## 4. Weekly digest (шлём ДАЖЕ при отсутствии событий)

- **Кадэнс:** раз в неделю, фикс. день/час (предложение — Пн 09:00 в таймзоне пользователя, дефолт UTC).
- **Пустая неделя — тоже письмо:** «За неделю материальных изменений нет. Кошелёк проверен N раз,
  последняя проверка <дата>, вердикт: <банда>.» Это **сигнал живости** (тишина ≠ «нас забыли»/«сервис умер»)
  и подтверждение, что мы реально сканируем.
- **Содержимое:** текущий вердикт-банд + `capital-under-elevated-risk $` + дельта за неделю (что появилось /
  ушло / изменилось, `computeReportDelta`) + топ-3 Watch, накопленные за неделю + «последняя проверка / next
  scan». Диплинк на полный отчёт.
- **Fail-closed в дайджесте:** если за неделю не удавалось проверить кошелёк — прямо это пишем
  («проверка устарела с <дата>, данные могли измениться»), не выдаём старый вердикт за свежий.

## 5. Формат сообщения (вердикт как в карточках чекапа)

Единый honest-tone. Вердикт — **posture-банд** (`clear / elevated / high` риск), НЕ прескриптивный приказ.
Структура каждого пункта — `факт → интерпретация → возможный эффект` (без «suggested direction», чтобы в
пуше не было даже намёка на совет-к-сделке; направление — уже в самом отчёте).

### 5.1 Telegram (CRITICAL, пример-скелет — числа подставляет движок)

```
🔴 DeFi Checkup — материальное изменение
Кошелёк …<wallet_ref[-4:]>  ·  вердикт: ПОВЫШЕННЫЙ РИСК

Что: <reason_code plain-language, шаблон> — например «stETH depeg-риск на held-позиции».
Капитал под повышенным риском: $X (было $Y, +$Δ за <период>).
Что это значит: <детерминированный шаблон по reason_code>.

Проверка: <дата/UTC> · policy <version> · доказуемо: /verify
[ Открыть отчёт ]   (inline-кнопка → deeplink §3)
```

### 5.2 E-mail (IMPORTANT/дайджест)

- **Subject:** `[DeFi Checkup] <банд>: капитал под риском $X (+$Δ)` — число и знак в теме, чтобы решать не
  открывая.
- **Тело:** тот же скелет + Critical/Important/Watch bucket, top-3 first + «что улучшилось» (RESOLVED) +
  секция «Источники и доказательства» (provenance, `output_hash`, last-verified, ссылка `/verify`) +
  **footer:** «Это не инвестиционный совет и не оферта. Мы измеряем риск, не держим ключи.» + one-click
  unsubscribe + «настроить пороги».
- **RU|EN двуязычно** (натуральный русский, без транслита — правило site-copy). Язык — по выбору пользователя
  при подписке.
- **HTML parse_mode-гигиена:** для Telegram — `parse_mode=HTML` (Markdown ломается на `_` в адресах/числах,
  известный баг SPA reporting).

### 5.3 Что НИКОГДА не попадает в сообщение

- Composite health-score (68/100) · вероятности сценариев · прескриптивные $-приказы («продай $170k») ·
  кнопки execute/Safe-tx · выдуманные APY/TVL/адреса · сырой адрес кошелька / PII · любые «minimum
  investment / no lock-up» формулировки (no-solicitation, продукт non-custodial).

## 6. Архитектура (эскиз, для оценки объёма — не финальный код)

```
report_run (движок пересчитал кошелёк)
        │  computeReportDelta(prev, curr)
        ▼
notification_evaluator (ДЕТЕРМИНИРОВАННЫЙ, без LLM)
        │  → классифицирует событие (CRITICAL/IMPORTANT/WATCH/RESOLVED/DATA-STALE)
        │  → сверяет с per-user порогами + дебаунс/гистерезис (persisted state)
        ▼
notification_queue (idempotent: dedupe по (wallet_ref, report_id, reason_code, class))
        │
        ├─► telegram_dispatcher  (мгновенно для CRITICAL/opt-in IMPORTANT)
        └─► email_dispatcher (Resend)  ⛔ gated на RESEND_API_KEY (own-07)
        │
weekly_digest_job (cron): собирает накопленное + «пустую неделю» → email (+opt-in TG)
```

- **Идемпотентность** — обязательна (перезапуск джобы не должен слать дубль). Ключ дедупа + журнал
  отправленного.
- **Хранилище состояния подписок/порогов/дебаунса:** БД checkup (`SPA_ACADEMY_DB`-стиль отдельный файл/pg на
  Railway). Атомарные записи.
- **Наблюдаемость:** счётчик отправленных/подавленных/ошибок (own privacy-counter стиль, без PII), чтобы
  видеть «шлём ли вообще» и «не спамим ли».

## 7. Что уходит в бэклог (и почему)

| Элемент | Причина отсрочки |
|---|---|
| **Browser/mobile push** | требует Web-Push/APNs-инфраструктуры + Service Worker; ценность < e-mail+TG на текущей стадии. Ждёт **PVA** (product-value validation). |
| **Webhooks (B2B)** | нужен только при **наклонной B2B / treasury-интеграции**; требует auth-контракта, ретраев, signature-подписи payload. Строить под конкретного клиента, не спекулятивно. |
| **Per-user Risk-Policy пороги (полный `#16`)** | большая фича (UI + evaluate-against-policy); пороги-дефолты закрывают MVP. Ждёт спроса treasury-покупателя. |
| **SMS** | дорого, низкий сигнал/шум для DeFi-аудитории; не планируем. |
