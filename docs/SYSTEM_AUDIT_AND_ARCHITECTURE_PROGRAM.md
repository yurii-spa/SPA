# Программа: полный аудит системы + архитектурная память («швейцарские часы»)

> **Статус:** ПРОГРАММА на утверждение владельца (owner-vision 2026-07-16). Большой многонедельный
> труд — здесь структура и **МЕХАНИЗМ**, как сделать это правильно и недеструктивно.
> **Цель владельца:** чтобы система была как швейцарские часы, а Claude заходил не «новым сотрудником
> с нуля», а как **CEO + архитектор-инженер**, который понимает и ПОМНИТ, почему всё устроено так.

## 0. Принципы (железные)

- **Недеструктивно.** Ничего не удаляем. Подозрительно-ненужное → **карантин** `attic/` с манифестом
  (откуда, когда, почему подозреваем), полностью обратимо. Удаление — только отдельным owner-решением позже.
- **Инкрементально.** Не big-bang. По одной подсистеме за проход; карта растёт, владелец видит прогресс.
- **Доказательно.** Каждый вердикт «живой / мёртвый / сломан-но-нужен / нужен-но-не-работает» — со
  свидетельством (импортируется? есть тесты? упоминается в агентах/cron? последний коммит?).
- **Память — источник правды в git.** Итог каждого прохода → durable-доки + Claude-memory. Obsidian —
  окно в них (можно использовать как рабочий слой), но правда в файлах.
- **Инварианты не трогаем.** Аудит read-first; любое перемещение/переименование — после карты и с
  проверкой, что не ломает runtime (RiskPolicy, go-live трек, деплой).

## 1. Механизм (как осилить огромный объём правильно)

**Подсистема → архитектор-агент → структурированный инвентарь → синтез в карту → карантин лишнего.**

1. **Разбиение на подсистемы** (единица одного прохода):
   `spa_core/{adapters,risk,governance,paper_trading,strategy_lab,swarm,monitoring,execution,api,
   reporting,telegram,owner_queue}` · `landing/` (сайт: страницы вкл. скрытые + admin) · `launchd/`+`scripts/`
   (флот агентов) · `data/` (state-файлы) · `docs/` · корень репо · **вне репо:** GitHub (репозитории,
   workflows, ветки), Railway (checkup/academy), Cloudflare (Pages/tunnel/Access).
2. **Один проход = один архитектор-агент** читает подсистему и выдаёт СТРУКТУРИРОВАННЫЙ инвентарь:
   что это · зачем существует (роль) · живой/мёртвый/сломан-но-нужен · кто использует (импорт/агент/cron/сайт) ·
   зависимости · проблемы именования/дублей · вердикт + рекомендация (оставить / переименовать / карантин / чинить).
3. **Синтез:** из инвентарей — растущая **КАРТА СИСТЕМЫ** (`docs/SYSTEM_MAP.md`): что где, почему так,
   как связано. Это и есть «память CEO/архитектора».
4. **Карантин лишнего:** подтверждённо-ненужное → `attic/<подсистема>/` + строка в `attic/MANIFEST.md`
   (откуда, дата, причина, как вернуть). НИКОГДА не удаляем на этом этапе.
5. **Целевая архитектура:** после карты — предложение по хранению/именованию (конвенции, целевая
   структура) → применяем инкрементально, отдельными проверенными шагами.

## 2. Рабочие потоки (workstreams)

- **WS-A · Файловый аудит репо.** Прочитать все подсистемы, инвентарь каждого файла, карантин мёртвого,
  вычистить дубли/junk (напр. известный «object-path junk», stale-скрипты), предложить именование/структуру.
- **WS-B · Аудит инфраструктуры.** GitHub (репо/workflows/ветки — что живёт, что заброшено); флот агентов
  по ролям — **разработка/research** (orchestrator, novel-edge-rnd, tournament…), **вклады/аллокация**
  (daily_cycle, allocator, swarm-книги), **мониторинг** (health, RTMR, freshness, watchdog); что дублируется,
  что RETIRED, что зомби. Railway (checkup/academy), Cloudflare (Pages/tunnel/Access).
- **WS-C · Аудит сайта.** Все страницы вкл. **скрытые** (noindex, orphaned) + **admin-панель** + access
  control: что публично, что внутреннее, что осиротело, зачем каждая. Карта сайта + «что лишнее/сырое».
- **WS-D · Архитектурная память.** `docs/SYSTEM_MAP.md` + обновление Claude-memory + `PROJECT_CONTROL/` так,
  чтобы любая сессия стартовала «как CEO», а не с нуля. Obsidian-vault как рабочее окно.

## 3. Как это исполняется (кадэнс)

- **Атрибутировано агентам + attended-ревью.** Каждый проход подсистемы — архитектор-агент (read-only
  инвентарь) → я синтезирую в карту → карантин/переименование с проверкой. По 1-2 подсистемы за заход.
- Автономный оркестратор (ежечасно) может брать по одному WS-A-инвентарю за цикл (read-only, безопасно);
  перемещения/карантин — только после карты и с тестами.
- Владелец видит растущую `SYSTEM_MAP.md` и карантин-манифест; правит приоритеты.

## 4. Deliverables

1. `docs/SYSTEM_MAP.md` — живая карта: каждая подсистема/файл/агент/страница — что, зачем, статус.
2. `attic/` + `attic/MANIFEST.md` — обратимый карантин лишнего.
3. Предложение по именованию/структуре (отдельный ADR перед применением).
4. Обновлённая Claude-memory + `PROJECT_CONTROL/` — «швейцарские часы»-контекст на старте сессии.
5. Инвентари по WS-B/WS-C (агенты, GitHub, сайт/admin) с пометками «не так / лишнее».

## 5. Owner-gates (решить перед стартом)

1. **Приоритет WS:** с чего начать — файлы репо (WS-A), инфра/агенты (WS-B), сайт/admin (WS-C) или память (WS-D)?
2. **Где карантин:** `attic/` в репо (git-виден, обратимо) — ок? (рекомендую да.)
3. **Глубина:** «прочитать всё» — это недели агент-проходов; ок тратить на это токены пачками (у тебя MAX)?
4. **Obsidian:** дать мне активно писать в vault как рабочий слой, или только git-доки + vault-зеркало?

_Программа — дизайн-документ. Исполнение WS начинается по твоему приоритету (§5). Ничего не удаляем;
всё обратимо; карта и память растут инкрементально._
