## 1. Финансовая модель: путь к $1M/год и $100M

### 1.1 Честная математика (исправление ТЗ)

Формула оценки: `Valuation = ARR × Multiple + NAV собственного капитала`.

- $1M ARR × 30–40x = **$30–40M**, не $100M. Для $100M нужно **$2.5–4M ARR**,
  т.е. **$100–200M внешнего AUM** при blended take rate 1.5–2%
  (3% take rate из ТЗ — выше рыночного для stable-vaults; закладывать нельзя).
- Доход на собственном капитале мультипликатор не получает — он стоит ≈ NAV.
  Собственный капитал — это **средство построить трек**, не бизнес-модель.

### 1.2 Базовый сценарий (план, по которому работаем)

| Период | Свой капитал (live) | Внешний AUM | Revenue | Комментарий |
|---|---|---|---|---|
| **2026 Q3** | $0 → $10–50K пилот | 0 | ~$0 | 30+ дней paper-трека (готов ~2026-07-10), E2E fork-harness, Safe |
| **2026 Q4** | $250K–1M | 0 | $5–20K (yield) | Live-исполнение доказано, 90+ дней трека, vault в разработке + аудит #1 запущен |
| **2027 Q1** | $0.5–1M | $1–3M (DAO-пилоты, ангелы) | run rate $60–120K | ERC-4626 запущен ПОСЛЕ аудита; mgmt fee 1.5% + perf 15% |
| **2027 H2** | $1–2M | $10–20M | run rate **$300–600K** | 12 мес трека → маркетинг трека, white-label API первые клиенты |
| **2028 H1** | $2–3M | $40–60M | **ARR пересекает $1M** | институциональный тираж; perf fee на high-watermark |
| **2028 H2 — 2029** | $3M+ | $100–200M | $2.5–4M ARR | **оценка $80–150M** при росте AUM >10%/мес и 0 инцидентов |

### 1.3 Агрессивный сценарий (из ТЗ — отслеживаем, не обещаем)

Vault уже в 2027 Q1 с $5–10M AUM и $1M ARR в 2027 — возможен только если:
(а) аудит без критических находок с первого раза, (б) первые DAO-казначейства
заходят без 12-месячного трека (бывает при сильном нарративе), (в) рынок
ставок не сжался. Вероятность ≤25%. План ресурсов строим по базовому сценарию,
агрессивный — upside.

### 1.4 Unit-экономика на $50M AUM (контрольная точка $1M ARR)

```
Mgmt fee 1.5% × $50M                  = $750K
Perf fee 15% × (7% yield × $50M)      = $525K
White-label API (3–5 B2B клиентов)     = $100–300K
────────────────────────────────────────────────
ARR                                    ≈ $1.3–1.6M
Расходы: аудиты+bounty $250K, инфра $50K, команда 2–3 чел $400–600K,
юр/комплаенс $100K  →  EBITDA-маржа 30–50%
```

### 1.5 Деревья смерти (kill-criteria, наследуются из GRAND_VISION §5)

- 2026-09-01: paper APY < 5% или gap в треке → live не больше $50K, пересмотр источников доходности.
- 2027-Q1: критические находки аудита → внешний капитал заморожен до re-аудита.
- 2027-Q3: AUM < $3M при готовом продукте → проблема distribution, разворот в white-label-first.
- Инцидент ≥1% AUM → стоп привлечения, публичный post-mortem, минус ступень Capital Ladder.

## 2. Суб-агентная архитектура (полная спецификация)

Конституция (не меняется никогда): **LLM не касается капитала.** Агенты
производят proposals; детерминированное ядро (L2 Risk + L3 Execution)
валидирует и исполняет. Граница оформляется CI-линтом (MP-309): запрет
import LLM-клиентов в `risk/`, `execution/`, `allocator/`.

### 2.1 Реестр агентов

| Агент | LLM | Читает | Решает / производит | Триггер | Логирует в | ЗАПРЕЩЕНО |
|---|---|---|---|---|---|---|
| **CEO** | топ-модель (Opus/Sonnet-класс) | equity_curve_daily, strategy_comparison, market regime, ceo_decisions (история), red_flags | какую стратегию активировать, когда менять allocation model, эскалации Owner'у, приоритизация бэклога | еженедельно (strategic loop) + немедленно при drawdown >2% | `data/ceo_decisions.json` (решение + reasoning + входной snapshot_id) | менять Risk Policy, инициировать tx, переопределять вердикт Risk |
| **Risk Sentinel** | детерминированное ядро + малая модель ТОЛЬКО для классификации алертов («шум / деградация / инцидент») | current_positions, risk_scores, live TVL/APY, depeg-фиды | флагует нарушения лимитов, инициирует kill-switch (детерминированная ветка), приостанавливает цикл при stale-данных | каждый цикл оркестратора (fast loop) | `data/risk_alerts.json`, alert_log | исполнять сделки; LLM-ветка не может менять вердикты детерминированной |
| **Alpha** | средняя модель (Flash/Haiku-класс) | DeFiLlama полный реестр, candidate-tier гейты, capacity-данные, конкурентный ландшафт | список кандидатов в whitelist + обоснование (APY-устойчивость, ёмкость, риск-оси) | еженедельно (slow loop) | `data/alpha_candidates.json` | добавлять протоколы в whitelist напрямую (только candidate-tier) |
| **Protocol Research** | топ-модель, редкие вызовы | досье кандидата: аудиты, команда, TVL-история, инциденты, governance | due-diligence отчёт к ADR на листинг | по запросу CEO/Alpha (кандидат прошёл гейты) | `data/research_dossiers/` | финальное решение о листинге (только человек/ADR) |
| **Strategy** | средняя модель | historical_apy, shadow-результаты S0–S5, backtest-метрики | параметры стратегий, Kelly-sizing предложения, рекомендация selector'у | еженедельно (slow loop) | `data/strategy_recommendations.json` | менять активную стратегию сам (исполняет селектор по детерминированному правилу) |
| **Reporting** | средняя модель | equity curve, trades, positions, decision log | ежедневный P&L (09:00, Telegram/email), monthly investor PDF (1-е число), объяснение решений человеческим языком | ежедневно 09:00 + 1-е число месяца | `data/reports/` + `latest_report.json` | искажать цифры: источник только данные, числовые поля проходят шаблонную валидацию против сырых JSON |
| **Incident Commander** | средняя модель | alert_log, incidents, system health, decision log | таймлайн инцидента, черновик post-mortem, чеклист реагирования | при CRITICAL алерте | `data/incidents/` | исполнять «исправляющие» транзакции |
| **Execution** | ❌ строго детерминированный | одобренный L2 AllocationTarget | diff-план → simulate → sign → confirm → reconcile | каждый цикл после Risk-одобрения | trades.json, decision log (tx hash) | любая LLM-логика |

### 2.2 Три контура координации

```
FAST LOOP    (каждый цикл, БЕЗ LLM):  data → Risk Sentinel(детерм.) → allocator → execution
             работает даже при полной недоступности LLM — требование автономности 95%
SLOW LOOP    (ежедневно, LLM):        Alpha-скан, Risk-классификация алертов, Reporting
STRATEGIC    (еженедельно, LLM):      CEO-синтез, Protocol Research, Strategy-тюнинг,
                                      предложения изменений ТОЛЬКО через ADR
```

Деградация: LLM недоступен → fast loop продолжает с последней одобренной
конфигурацией; ничего нового не листится, лимиты не меняются, отчёты
откладываются в очередь.

### 2.3 Контракт аудируемости (единый для всех агентов)

Каждый вызов агента пишет запись: `{agent, ts, input_snapshot_id, model,
tokens, proposal, deterministic_verdict, final_action, correlation_id}`.
Цепочка snapshot → proposal → risk-вердикт → trade/tx восстановима для
любого решения (MP-310). Это и есть продукт для институционала.

## 6. Что мы сознательно НЕ делаем (и почему)

- **Leverage/looping ради APY 15%+** — ломает риск-дисциплину, единственный moat.

  в первом же документе — самоубийство нарратива «honesty by design».
